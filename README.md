## 1. サービス概要

- このアプリは看護師向けの勤務シフト作成アプリです
　
- シフト作成者の負担となるシフト作成時間の軽減や
　連続勤務や夜勤の連続などのスタッフから不満の出やすい勤務形態の回避を目的としています

- ロジックを組み込んだシフトの自動作成にて上記の問題を解決します

---

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

看護師のシフト作りはほぼ病棟師長が行なっています
私の勤務していた病棟では師長がシフト作りに難渋して、毎月シフトの出来上がりが前月末ギリギリになることが多くありました

シフト作りに対して
師長は「シフト作りはメインの業務じゃない割に時間がかかるから手につかない...」
メンバーは「来月も予定があるから早くシフト出してほしいし、極端な連勤とか止めてほしい...」
と重要度の乖離があって、ここがスムーズにいかないと軋轢を生みやすい部分でした

ある程度ロジックに則ったシフトの自動作成ができれば、これらの問題が解決できるのでは？と思い、このアイデアが生まれました

---

### 2-2. なぜそれが「気になった」のか

実際に自分が「ここどうにかうまいことならないかな...」と感じていた課題であり

・時間がかかる業務であること
・ロジックで再現できそう
・師長、メンバーと広範囲に恩恵が出る

これらの理由から自動化に適している課題だと考えたから

## 7. このアプリで実現すること
### MVPで作る機能
- スタッフ登録機能
  - スタッフ名
  - 夜勤可否
  - 勤務レベル（まずリーダー、メンバーのみ）
  - 月の勤務上限・夜勤回数・休みの回数

- 希望休登録機能
  - 各スタッフの希望休を登録できる

- 必要人数設定機能
  - 日ごとの日勤人数を指定できる
  - 日ごとの夜勤人数を指定できる

- シフト自動作成機能
  - 日勤、夜勤、明け、休みを自動で割り当てる
  - 夜勤明けの次の日は休みにする、希望休には勤務を入れないなどの禁忌条件の登録
  - 指定した人数を満たすようにシフト案を生成する

- シフト確認機能
  - 月単位のシフト表を画面で確認できる
  - 条件違反がある場合は警告を表示する

### 本リリースで作る機能

- シフト作成者とシフト入力者でユーザーの権限と画面を分ける
  - 入力者は希望休入力機能、レビュー機能
  - 作成者は作成画面

- シフト生成精度の向上
  - 条件が厳しすぎて配置できない場合に、どの条件が原因か表示する
  - 禁忌ルールを一部緩和して再生成できるようにする

- 条件設定の追加
  - 勤務レベル別の人数配置
  - リーダー可能者を各勤務に配置
  - 男性スタッフを各勤務に1人以上配置
  - 新人のみの勤務にならないようにする
  - 連勤数の上限を設定する
  - 夜勤回数の偏りを減らす

- 導入しやすいUI
  - PC操作が苦手な人でも使える入力画面（できればExcel風に）
  - 操作手順がわかるチュートリアル
  - 入力ミスを防ぐフォーム設計
  - CSVインポート・エクスポート
  - 希望休公開モード、非公開モードの追加

- シフト満足度レビュー機能
  - 完成したシフトに対してスタッフ満足度を記録する
  - 不満が出やすい勤務パターンを分析する
  - 次回以降のシフト生成に活かす

### 10-2. 技術選定の理由

- なぜこの技術を使うのか

Djangoは、ユーザー認証、管理画面、フォーム、DB操作など、Webアプリに必要な機能が一通り揃っているため、短期間でMVPを作りやすいと考えました。
このアプリでは、スタッフ情報、希望休、勤務条件、作成済みシフトなど、DBで管理するデータが多くなります。DjangoのModelを使うことで、これらのデータ構造を整理しながら開発できると考えました。
PostgreSQLは本番環境でも利用しやすく、Renderとの相性も良いため採用します。

- 今回チャレンジしたい点

今回チャレンジしたい点は、単なるCRUDアプリではなく、条件に基づいてシフトを自動生成する業務ロジックを実装することです。
具体的には、以下のような条件を扱います。
- 希望休には勤務を入れない
- 夜勤明けの次の日は休みにする
- 日ごとに必要な日勤者・夜勤者の人数を満たす
- 夜勤回数が偏りすぎないようにする
- 勤務レベルやリーダー配置を考慮する

これらの条件をもとに、ユーザーが修正しやすいシフト案を自動生成できるようにすることが、このアプリで一番挑戦したい部分です。

- 不安な点

一番の不安は、シフト自動生成ロジックの難易度です。
シフト作成では、希望休、必要人数、夜勤回数、連勤制限、勤務レベルなど、複数の条件を同時に考慮する必要があります。
人数が少ない場合や希望休が多い場合は、すべての条件を満たすシフトが作れない可能性もあります。
そのため、MVPでは最初から完璧なシフトを作るのではなく、最低限の条件を満たした「修正しやすいたたき台」を作ることを目標にします。
本リリースでは、条件違反の警告表示、ルールの優先度設定、条件を緩和した再生成などを追加し、段階的に精度を上げていきたいと考えています

### 10-3. キャッチアップ不足の懸念

キャッチアップ不足の段階では、以下の懸念点があります。

1. シフト自動生成ロジックの難易度が高い

このアプリの中心機能は、勤務条件をもとにシフトを自動生成する部分です。
通常のCRUDアプリと比べて、条件分岐や組み合わせの考え方が複雑になるため、実装難易度が高いと考えています。

対策として、MVPでは以下のように条件を絞って実装します。

- 日勤、夜勤、明け、休みの割り当て
- 夜勤明けの次の日は休み
- 希望休には勤務を入れない
- 日ごとの必要人数を満たす

その後、勤務レベルやリーダー配置、連勤制限などの複雑な条件を段階的に追加します。

１. React / Next.js の導入について

ReactやNext.jsは、シフト表のような複雑なUIを作る場合に有効ですが、卒業制作の限られた期間でDjangoに加えてReactまで本格的に導入すると
キャッチアップコストが高くなる可能性があります。

そのため、MVPではDjangoテンプレートを中心に実装し、本リリース後の改善としてReactの導入を検討します。
まずはDjangoで一通り動くアプリを完成させ、必要に応じてシフト表画面やドラッグ操作部分だけReact化する方針にします。
